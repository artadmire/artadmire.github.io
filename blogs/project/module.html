    # 模块化

    # 最初

    JS做的工作相对简单,就是一些表单提交,页面动画。

    # 需求出现

    JS的模块化需求日益增长

    文件分离推动了模块化的发展。

    1. 开始需要在页面增加不同的js，如：动画，表单，格式化
    2. 开始使用多个js文件
    3. 不同的文件又被同一模板引用
    <script src="jquery.js"></script>
  <script src="main.js"></script>
  <script src="dep1.js"></script>
  //……

    象一下,如果多个js文件交给多人开发,最后又出现在一个页面,问题开始慢慢出现...

    多文件的出现暴露出下面几个问题：

    • 污染全局作用域
    • 维护成本高
    • 依赖关系不明显

    # 雏形 - IIFE

    使用"立即执行函数"（Immediately-Invoked Function Expression，IIFE）。其实就是对作用域的把控 ,可以达到不暴露私有成员的目的。

    定义了一个函数，如果立即执行，多个函数之间就隔离开了

      (() => {
    let count = 0;
    // ……
  }();

    尝试去定义一个最简单的模块,暴露一些方法。

    const iifeModule = (
  function() {
    let count = 0;
    const add = () => count ++;
    const reset = () => 0;
    return  {
      add,
      reset
    }
  }
)();
iifeModule.add();

    • 暴露了方法
    • 在一定程度上解决变量污染

    问题：有依赖时，如何解决？

    参数形式传入。

    const reset = () => 0;

      reset
    }
  }
)(module1, module2);

    # Commonjs规范

    node.js中使用 特征：

    • 通过module + exports 去对外暴露接口
    • 通过require来调用其他模块

    模块组织方式 main.js 文件

    // 引入部分
const dependencyModule1 = require(./dependencyModule1);
const dependencyModule2 = require(./dependencyModule2);

// 处理部分
let count = 0;
const increase = () => ++count;
const reset = () => {
  count = 0;
}
// 做一些跟引入依赖相关事宜……

// 暴露方式1
exports.increase = increase;
exports.reset = reset;

// 暴露方式1
module.exports = {
  increase, reset
}

    • 优点： CommonJS率先在服务端实现了，从框架层面解决依赖、全局变量污染的问题

    commonJs规范导出和引入总结：

    • module.exports 初始值为一个空对象 {}，所以 exports 初始值也是 {}； module.exports === exports 为true
    • exports 的改变影响不了module.exports。
    • require() 返回的是 module.exports 而不是 exports；

    # AMD规范

    通过异步加载 + 允许制定回调函数

    经典实现框架是：require.js

    与commonJS 也有一些相似之处，commonJS 暴露给外部的接口为module.exports 所指向的对象，而AMD 暴露给外部的接口为return 的对象。

    新增定义方式:

      // 通过define来定义一个模块，然后require进行加载
  /*
  define
  params: 模块名（可选），依赖模块（可选），工厂方法
   */
  define(id, [depends], callback);
  require([module], callback);

    # UMD(跨平台)

    • AMD模块以浏览器第一的原则发展，异步加载模块。
    • CommonJS模块以服务器第一原则发展，选择同步加载。
    • 通用的模式UMD （Universal Module Definition）。希望解决跨平台的解决方案。
    (function (window, factory) {
    if (typeof module === 'object' && typeof exports === 'object') {
     
        module.exports = factory();
    } else if (typeof define === 'function' && define.amd) {
     
        define(factory);
    } else {
     
        window.eventUtil = factory();
    }
})(this, function () {
    //module ...
});

    # CMD规范

    按需加载 主要应用的框架 sea.js

      define('module', (require, exports, module) => {
    let $ = require('jquery');
    // jquery相关逻辑

    let dependencyModule1 = require('./dependecyModule1');
    // dependencyModule1相关逻辑
  })

    • 优点：按需加载，依赖就近
    • 缺点: 依赖于打包，加载逻辑存在于每个模块中，扩大模块体积

    总结：

    • 无模块化到IIFE是一个从无到有的过程；
    • IIFE到Commonjs是一个从代码层面优化到框架层面优化的过程；
    • Commonjs到AMD是一个从同步到异步的区别；
    • AMD到CMD是优化了按需加载，依赖加载的过程。

    # ES6模块化

    es6代码

    支持同时导出

    export const bar = 2;
export function foo() {}
const bar1 = 4;
function foo1() {}
export {bar1 , foo1}
const b = {}
export default b;

    引入：

    import {bar, foo, bar1, foo1} from './utils/index';
    1

    或者只引入默认导出：

    import b from './utils/index';
    1

    也支持同时引入：

    import b, {bar ,foo} from './utils/index';
    1

    ES6引入总结：

    在一个文件或模块中，export、import可以有多个，export default仅有一个 通过export方式导出，在导入时要加{ }，export default则不需要。
