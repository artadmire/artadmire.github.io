    搭建一个组件库

    maile

    # 搭建一个组件库

    # 开始

    目标：一个仓库内管理多个项目，以多个npm包方式发布，依赖集中管理，npm包版本可以集中管理，也可以单独管理。通常适用于有一定关联的组件，但各组件需要支持单独的npm包发布和安装。

    优点：

    • 共同依赖统一管理，版本控制更加容易，依赖管理会变的方便。
    • 支持组件的单独发布和单独构建。
    • 使用时可以单独引入。

    缺陷：

    • 项目搭建复杂度高。

    需要做什么？

    • 代码验证
    • 单元测试 jest
    • 构建rollup & webpack
    • 支持ts
    • 提交卡点
    • 组件库文档编写

    # 前言

    诸多选择以后，我觉得选择pnpm搭建自己的组件库。

    理由如下：

    • lerna 已经不再维护，后续有任何问题社区无法及时响应。

    • pnpm 非常适合「单仓多包」的项目，即 monorepo。pnpm 不仅速度够快，占用的空间更小，尤其是 monorepo 项目，一个仓库下的多个项目可共用依赖包，各个项目可通过软链索引到依赖包的实际位置（类似文件夹快捷方式），而不需要每个项目都安装一份，大大节省了存储空间。

    • pnpm 的使用命令和 npm 类似，迁移学习成本低，基本看一遍官方文档就能上手。

    • 既然大家都说它很好用，不去尝试一下怎么行呢？

    # 安装依赖

    1. 根目录下安装
    pnpm install typescript -D -W
    1
    2
    1. 单独安装

    比如我们只在 @maile/ui 应用中用到react，那就可以为它单独安装。首先要拿到它的 package name，这个是我们在子项目中自定义的，那么命令如下：

    pnpm install @maile/utils -r --filter @maile/ui

pnpm i -r --filter vite-ts

    1. 相互依赖

    在 monorepo 中，我们往往需要 package 间的引用，比如本例中的@maile/utils就会被@maile/ui依赖，或者还有maile-cli。

    pnpm i @maile/utils -r --filter @maile/ui @maile/cli
    1
    2
    3

    这个时候就可以在@maile/ui中使用了:

    import { debounce } from '@maile/utils';

    2

    当执行了pnpm publish后，会把基于的workspace的依赖变成外部依赖，如：

"dependencies": {

  },
// after
"dependencies": {
    "@maile/utils": "^1.0.0"

    # 开发

    一个 monorepo 往往是一个整体的项目，所以我们需要同时执行多个指令，在 pnpm 中，可以通过-C进行配置：

    
"scripts": {
  "build": "pnpm -C ./packages/maile-ui build ",
}

    # peerDependencies

    在主系统的 package.json 中的 dependencies 声明下 react 和 react-dom 的版本：

    主系统package.json:

    {
  "dependencies": {
    "react": "^16.13.1",
    "react-dom": "^16.13.1"
  }
}

    组件库的package.json：

    {
  "peerDependencies": {
    "react": ">=16.12.0",
    "react-dom": ">=16.12.0"
  }
