    一. src和href的区别

    maile

    # 一. src和href的区别

    个人觉得有两点比较重要:

    1. 当浏览器遇到href会并行下载资源并且不会停止对当前文档的处理。由css解析器进行解析，阻塞浏览器渲染。能够解决闪屏问题。(同时也是为什么建议使用 link 方式加载 CSS，而不是使用 @import 方式)。

    2. 当浏览器解析到src ，会暂停其他资源的下载和处理，直到将该资源加载或执行完毕。(这也是script标签为什么放在底部而不是头部的原因)

    # 二. script标签中defer和async的区别

    js的下载和执行都会阻塞浏览器的渲染，所以如果把js的引入放在body前面，浏览器遇到script标签，就会先去下载资源，再执行js文件，之后再继续解析渲染页面。如果你想把script标签放在前面，又不阻塞dom的解析和渲染，就可以使用defer或者async解决，但是二者又有差别。

    相同点： js的加载不会阻止dom的解析。

    不同点：

    1. 使用defer，紧随页面dom解析之后，在DOMContentLoaded事件触发之前执行。特点如下:
    • 不会阻塞dom的解析，加载完成后会等待dom解析完成再执行
    • 多个标签都使用defer，会安装出现的先后顺序加载执行。即使有的先加载完成，也会按照顺序执行。
    1. 使用async，在window的load事件触发之前执行完成。特点如下：
    • 加载完就执行，如果dom没有解析完成，就会阻塞dom解析，先执行js。
    • 多个标签都使用async，谁先加载完，谁先执行，不能保证顺序。

    # 三. iframe有哪些优缺点

    优点：

    • iframe能够原封不动的把嵌入的网页展现出来。
    • 天然的css和js隔离
    • 可复用性强

    缺点：

    • iframes阻塞页面加载，影响网页加载速度。
    • 如果页面有弹框，弹框只会出现在iframe内
    • 浏览器刷新之后 iframe url 状态丢失，后退或者前进按钮无法使用。
    • 上下文隔离，数据不共享

    # 三. 渐进增强和优雅降级有什么不同？

    渐进增强和优雅降级这两个概念是在 CSS3 出现之后火起来的。由于低级浏览器不支持 CSS3，但是 CSS3 特效太优秀不忍放弃。所以产生了的一种解决方式在高级浏览器中使用CSS3，而在低级浏览器只保证最基本的功能。

    何谓渐进增强：

    渐进增强：一开始就针对低版本浏览器进行构建页面，完成基本的功能，然后再针对高级浏览器进行效果、交互、追加功能达到更好的体验。

    何谓优雅降级：

    优雅降级：一开始就构建站点的完整功能，然后针对浏览器测试和修复。比如一开始使用 CSS3 的特性构建了一个应用，然后逐步针对各大浏览器进行 hack 使其可以在低版本浏览器上正常浏览。

    这样看来，两者应该是工作流程上的不同，或者说侧重点不同，解决问题的思路不同。

    渐进增强是针对低版本浏览器来设计，优雅降级侧重点在高版本的浏览器。

    # 五. 位什么用transform代替position来改变位置

    简单来说就是使用position来完成动画，会使浏览频繁的重绘和重排，如果动画过于频繁，就会造成浏览器的卡顿，而使用transform，则能很多的解决这个问题。

    优化方案就是利用GPU硬件加速。浏览器渲染dom的时候简单步骤如下：

    分割图层--样式计算--重绘，重排--图层上传GPU--重组图层。

    transform的相关属性会形成复合层，开启硬件加速，这样就省去了中间重绘重排的过程，最后是图层的重组。

    # 七. undefined和null的区别

    1. typeof返回的类型不同，一个是undefined，一个是object

    2. null 指空对象，但它是被定义过的，undefined 指声明未赋值的对象或者是不存在的对象属性值。

    3. undefined可以作为变量，null不能

    4. 转化为number的值不同

      console.log(Number(undefined)) // 0
  console.log(Number(null)) // NaN

    # 十一. computed和watch的区别

    1. computed计算属性，data中不存在，有缓存。不建议进行赋值，会造成胡数据流混乱
    2. watch监听属性，data中存在，无缓存，可以进行异步。

    # 十二. $nextTick的原理

    调用$nextTick函数接收一个回调函数，这个回调函数会被加入callbacks数组，这个数组的执行时机是所有的同步任务执行完以后才去执行，用微任务包裹。callbacks还会有watcher实例，并且会排在前面，执行以前会先执行watcher.run更新dom，但是页面还没有渲染，因为页面的渲染会在所有微任务执行完毕以后，所以$nextTick的回调函数可以拿到更新的dom和数据。

    # 有状态和无状态组件的理解和使用场景

    无状态组件：不需要维护自己的状态，可以有props也可以没有。

    有状态组件：有自己的state维护。

    使用场景：

    • 如果想要存储一些数据并且想要对这些数据进行增删改查那么就应该使用有状态的组件。
    • 如果说要展示一个列表，建议使用无状态组件，数据通过props传递。

    # HTTP1.0和HTTP2.0的区别

    • HTTP/1.* 一次请求-响应，建立一个连接，用完关闭；每一个请求都要建立一个连接；同个与域名下，浏览器对于建立TCP连接的个数有限制。

    • HTTP/1.1 解决方式为，若干个请求排队串行化单线程处理，后面的请求等待前面请求的返回才能获得执行机会，一旦有某请求超时等，后续请求只能被阻塞，毫无办法，也就是人们常说的队头阻塞；

    • HTTP/2多个请求可同时在一个连接上并行执行。某个请求任务耗时严重，不会影响到其它连接的正常执行；还支持头部压缩和服务端推送。采用二进制传输。

    # HTTP状态码304是多好还是少好？

    304状态码说明命中了缓存，首次加载页面肯定不会出现，再次加载我觉得是越多越好吧。毕竟加载缓存数据比向服务器发送请求加载数据要快的多。

    # 什么是xss攻击？

    Cross-Site Scripting（跨站脚本攻击）简称 XSS，是一种代码注入攻击。XSS 攻击通常指的是利用网页的漏洞，攻击者通过巧妙的方法注入 XSS 代码到网页，因为浏览器无法分辨哪些脚本是可信的，导致 XSS 脚本被执行。XSS 脚本通常能够窃取用户数据并发送到攻击者的网站，或者冒充用户，调用目标网站接口并执行攻击者指定的操作。

    1. websocket

    server.js

    // index.js
let WebSocketServer = require("ws").Server;
let wss = new WebSocketServer({ port: 3000 });
 
 
// 创建保存所有已连接到服务器的客户端对象的数组
let clients = [];
 
 
// 为服务器添加connection事件监听，当有客户端连接到服务端时，立刻将客户端对象保存进数组中。
wss.on("connection", function (client) {
  console.log("一个客户端连接到服务器");
  if (clients.indexOf(client) === -1) {
    clients.push(client);
    // 接收客户端发送的消息
    client.on("message", function (msg) {
      console.log("收到消息:" + msg);
      // 将消息发送给非自己的客户端
      for (let key of clients) {
        if (key != client) {
          key.send(msg.toString());
        }
      }
    });
  }
});

    # isNaN和Number.isNaN函数的区别？

    函数isNaN接收参数后，会尝试将这个参数转换为数值，任何不能被转换为数值的值都会返回true，因此非数字值传入也会返回true，会影响NaN的判断。

    函数Number.isNaN会首先判断传入参数是否为数字，如果是数字再继续判断是否为NaN，不会进行数据类型的转换，这种方法对于NaN的判断更为准确。
